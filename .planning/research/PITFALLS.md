# Domain Pitfalls

**Domain:** React class-to-hooks migration orchestrator + TypeScript strict mode hardening
**Researched:** 2026-04-01
**Sources:** React official docs (react.dev), ts-morph bridge codebase inspection, project codebase analysis

---

## Critical Pitfalls

Mistakes that cause incorrect output, silent regressions, or full rewrites of a migration phase.

---

### Pitfall 1: False-Positive "safe-auto" Classification for Components That Cannot Be Migrated

**What goes wrong:**
The triage/classification agent marks a component as "safe-auto" but the component contains one of several class-only patterns that have no hook equivalent:
- `getSnapshotBeforeUpdate` — React explicitly states there is no function component equivalent. A component using this lifecycle MUST remain a class component (or requires a fundamentally different DOM-management strategy).
- `componentDidCatch` / error boundaries — Error boundaries are class-component-only. There is no hook-based equivalent. The `react-error-boundary` library wraps a class component internally; auto-migrating an error boundary will break it silently or cause a runtime error.
- Instance variables used as mutable state that drives re-rendering — If `this.someVar = x` is followed by a forced re-render via `this.forceUpdate()`, converting to `useRef` will break the render (refs do not trigger re-renders), and converting to `useState` changes the semantics.

**Why it happens:**
AST skeleton inspection may not distinguish `getSnapshotBeforeUpdate` from other lifecycle methods that do have hook equivalents. The classification heuristic ("no lifecycle methods beyond the standard trio") is insufficient. The `forceUpdate()` pattern is especially deceptive — it looks like a mutation but drives rendering.

**Consequences:**
Generated code compiles (tsc gate passes because the shape is valid) but behaves differently at runtime. No test coverage catches it. The PR looks clean.

**Prevention:**
- Add an explicit blocklist check before classification: scan AST for `getSnapshotBeforeUpdate`, `componentDidCatch`, `getDerivedStateFromError`, and `forceUpdate`. Any match is an unconditional `manual` routing — not `assisted`, `manual`.
- Document these as unmigratable in v1; escalate to manual with a named explanation in the PR description.
- In the triage extension to `agentic/triage.py`, these are hard-coded `category="manual"` short-circuits before the LLM runs — not a confidence score.

**Detection (warning signs):**
- Component imports or calls `forceUpdate`
- Component defines `getSnapshotBeforeUpdate`
- Component class name contains "ErrorBoundary" or defines both `getDerivedStateFromError` and `componentDidCatch`
- Component defines `static contextType` combined with `componentDidCatch`

**Phase:** Triage/Classification phase (extending `agentic/triage.py`). Must be wired before any migration transform runs.

### Pitfall 2: Hook Ordering Violations Generated by Mechanical Transformation

**What goes wrong:**
Automated class-to-hooks transformation produces code that violates the Rules of Hooks — specifically, hooks called conditionally or after an early return. This is the single most common source of silent bugs from mechanical migration.

The specific patterns that cause violations:
- A method that was called conditionally in `render()` is converted to a `useEffect` or `useState` call placed inside an `if` block or after a conditional `return`.
- Class components with a guard pattern like `if (!this.props.data) return null;` placed before a hook call — when this guard is naively transplanted at the top of the function component, it causes hooks to be called after a conditional return.
- Hooks inside event handlers — the migration naively inlines `useContext` or `useState` inside an `onClick` handler because the original method accessed `this.state` inside an event handler.

**Why it happens:**
ts-morph AST transforms are syntactically correct but semantically wrong. The bridge (see `handlers.ts`) operates on the tree structurally without semantic awareness of hook execution order constraints. A naive "move this.state accesses to useState calls" will not respect conditional placement.

**Consequences:**
Runtime error: "React has detected a change in the order of Hooks called by [Component]." This only surfaces at runtime, not at tsc compile time. The tsc gate will pass.

**Prevention:**
- After transform, run `eslint-plugin-react-hooks` as a post-transform lint gate, not just as a development tool. Wire this as a separate validation step in the migration pipeline between transform and tsc gate.
- Build a pre-migration AST analysis step: before transforming, check whether the class's `render()` method contains conditional returns that occur before any method calls that will become hooks. Flag these as `assisted` at minimum.
- The validator in `agentic/validator.py` should be extended to run the hooks lint rule programmatically after each file transform.

**Detection (warning signs):**
- `render()` contains multiple `return` statements (not just the final one)
- `render()` accesses `this.state` or calls methods inside ternary branches or `&&` expressions
- Class has logic like `if (this.props.loading) return <Spinner />`

**Phase:** Transform phase and post-transform validation. Must also be a gate before `apply_changes()` writes to disk.

### Pitfall 3: tsc Error Count Gaming — Counting Errors Is Not the Same as Measuring Safety

**What goes wrong:**
The tsc before/after gate compares total error counts. A migration can pass the gate (zero count increase) while introducing new errors by eliminating pre-existing errors in the same files that had been suppressed or were of a different severity. More critically:

- A transform introduces new type errors in one file but the same transform removes syntax errors in another, leaving the net count unchanged or lower.
- `getPreEmitDiagnostics()` on the ts-morph `Project` includes declaration emit diagnostics as well as type errors. If the migration changes module exports, declaration diagnostics can spike or drop independently of type correctness.
- In the existing `handleGetDiagnostics` in `handlers.ts`, diagnostics are collected via `p.getPreEmitDiagnostics()`. This includes `ts.DiagnosticCategory.Suggestion` (category 3) and `ts.DiagnosticCategory.Message` (category 0) in addition to errors (category 1) and warnings (category 2). If suggestions increase but errors decrease by the same count, the gate silently passes.

**Why it happens:**
Error count is a proxy metric. Proxy metrics are gameable and don't capture the semantics of "did we make things worse in migrated files."

**Consequences:**
PR is opened, merged, and a type regression ships in the component that was migrated. The regression only surfaces when another engineer tries to use the component.

**Prevention:**
- Gate on error count by severity separately: count only `DiagnosticCategory.Error` (category 1). Do not mix warnings, suggestions, or messages into the pass/fail metric.
- Scope the comparison: compare errors in the files touched by the migration, not project-wide. A pre-existing error in an unrelated file should not absorb a new error introduced in a migrated file. Implement per-file error tracking in the before/after comparison.
- Require zero new errors in migrated files specifically, not zero net increase project-wide.
- Store the before-snapshot as a `frozenset` of `(file, line, code)` tuples, not just a count, so that moved errors are distinguished from new ones.

**Detection (warning signs):**
- Project has pre-existing errors before migration (tsc before-check shows non-zero)
- Multiple files are migrated in a single batch (net count averaging masks per-file regressions)
- Migration modifies re-exported types or barrel files (affects many downstream counts)

**Phase:** tsc gate design (v1 requirement: "tsc after-check as pass/fail exit gate"). The gate implementation in the executor must use per-file, error-only, fingerprint-based comparison from the start.

### Pitfall 4: State Batching Semantic Divergence — setState vs useState

**What goes wrong:**
Class component `setState` and hook `useState` have subtly different semantics when called multiple times in the same handler. A migrated component may appear correct but produce different state values:

```ts
// Class: each setState enqueues independently, final value correct
this.setState({ count: this.state.count + 1 });
this.setState({ count: this.state.count + 1 });
// Final count: class.state.count + 1 (batched, but reads same this.state)

// Naive hook migration:
setCount(count + 1);
setCount(count + 1);
// Final count: count + 1 (NOT count + 2 — stale closure)
```

The correct migration requires an updater function (`setCount(c => c + 1)`), but a mechanical transform will emit the naive version.

In React 18, automatic batching outside of event handlers (timeouts, promises, native event listeners) is a new behavior that breaks class components relying on synchronous state reads after `setState` in those contexts. The migrated hook version will batch where the class version did not.

**Why it happens:**
AST-level transform copies the value reference (`this.state.count`) into the hook call without detecting that the new code is inside a closure with stale capture semantics.

**Consequences:**
Off-by-one bugs in counters, incorrect accumulated state in form handlers, subtle race conditions in async handlers. These pass tsc and may pass unit tests that only check final rendered output.

**Prevention:**
- The migration transform must detect multiple `setState` calls on the same state key within the same handler (synchronous code block) and emit updater-function form `setState(prev => ...)` rather than direct value form.
- Add a static analysis check: if the same state property is written more than once in the same method body, flag the component as `assisted` rather than `safe-auto`.
- In the classification heuristic, methods with multiple `setState` calls on the same key count as one medium-complexity signal.

**Detection (warning signs):**
- `setState` called more than once in the same method
- `setState` called inside `setTimeout`, `Promise.then`, or native event listeners
- `setState` called with a value that reads `this.state.*` in the same expression

**Phase:** Both classification (flag as assisted if pattern detected) and transform logic.

### Pitfall 5: getDerivedStateFromProps Produces Incorrect Hook Equivalent

**What goes wrong:**
`getDerivedStateFromProps(props, state)` fires on every render (not just prop changes) and has no direct hook equivalent. Migrations that attempt to replace it with `useEffect` introduce a timing bug: the effect runs after the render, whereas `getDerivedStateFromProps` runs before render, synchronously. The component will render once with stale derived state, then update — causing a flash or incorrect intermediate state.

The deeper problem: `getDerivedStateFromProps` is often used as a controlled/uncontrolled component reset pattern (`if (props.userId !== state.prevUserId) return { email: props.defaultEmail }`). The correct migration is to change the component's architecture (fully controlled, or use a `key` prop to reset), not to transplant the lifecycle call.

**Why it happens:**
The method is static and has access to both props and state, which makes it look like a simple mapping function. But the semantics (synchronous, pre-render, every-render) cannot be replicated with any hook.

**Consequences:**
Visible render flickering in controlled form components. Incorrect state shown to user for one render frame before correction. Potential infinite re-render loops if the effect sets state that triggers another render.

**Prevention:**
- Any component with `getDerivedStateFromProps` is unconditionally escalated to `assisted` or `manual` in classification. Do not attempt automated migration.
- In the PR description, document each component with this lifecycle as needing architectural review, not just a transform.
- If migration is attempted in a future version, it requires a strategy decision (fully controlled? reset with `key`?) that the agent cannot make autonomously.

**Detection (warning signs):**
- Component defines `static getDerivedStateFromProps`
- Component state contains a `prev*` prop mirror field (e.g., `prevUserId`, `prevDataProp`)

**Phase:** Classification phase — hard `assisted` escalation, not a transform target in v1.

### Pitfall 6: Ref Migration Breaks Components Exposing Imperative Instance Methods

**What goes wrong:**
Class components frequently expose instance methods via refs (e.g., `myRef.current.focus()`, `myRef.current.reset()`, `myRef.current.scrollToTop()`). These are called by parent components. When the class component is converted to a function component, the ref type changes from the class instance (which has those methods) to whatever `useImperativeHandle` exports.

If the migration omits `useImperativeHandle` and `forwardRef` (or the React 19 direct ref prop pattern), parent components calling `ref.current.focus()` will get a TypeScript error or a runtime null dereference. Even worse: if the migrated component wraps a DOM element and the ref type changes from `MyComponent` to `HTMLInputElement`, the parent's explicit method calls will fail silently at runtime (the methods don't exist on the element).

Note: React 19 deprecated `forwardRef` in favor of passing `ref` as a regular prop. If the target codebase is React 18, `forwardRef` is required. If React 19, it is not. The migration transform must check the React version.

**Why it happens:**
AST analysis of the class component does not traverse the call sites of components using this component as a ref target. The ref interface contract is implicit — it is not encoded in the class definition itself.

**Consequences:**
TypeScript errors in parent components (caught by tsc gate if the ref type is correctly typed). Runtime errors if the ref type is `any` or untyped (not caught by tsc gate).

**Prevention:**
- In the pre-migration AST scan, check whether the component is used with a `ref` prop at any call site in the project using `find_references`. If yes, escalate to `assisted`.
- Alternatively, check if the class has any public methods beyond `render()`. Public methods on a class component are almost always exposed via refs.
- When generating the PR description, explicitly list components with public instance methods as "ref contract components — manual review required."

**Detection (warning signs):**
- Class has public methods other than lifecycle methods and `render`
- Component name appears in `find_references` results with `.current.` access patterns

**Phase:** Classification phase (public method check) and PR description generation (list ref-contract components explicitly).

### Pitfall 7: useEffect Dependency Array Errors Are Not Caught by tsc

**What goes wrong:**
Missing or incorrect `useEffect` dependency arrays are the most common source of stale closure bugs after migration. These are not TypeScript type errors — tsc will not catch them. They require `eslint-plugin-react-hooks` with the `exhaustive-deps` rule enabled.

A migration that produces a valid `useEffect` with an incorrect dependency array (missing a prop or state variable that is read inside the effect) will pass the tsc gate but introduce subtle subscription, timer, or data-fetching bugs.

**Prevention:**
- Wire eslint with `react-hooks/exhaustive-deps` as a post-transform lint gate, not only as a developer tool. This must run before `apply_changes()`.
- The `DiagnosticEntry` model in the pipeline should be extended to include ESLint findings alongside tsc diagnostics for the pre/post comparison.
- In the PR description, report ESLint hook warning count alongside tsc error delta.

**Detection (warning signs):**
- `useEffect` body reads a prop or state variable not present in the dependency array
- `useEffect` has an empty dependency array `[]` but the body references props

**Phase:** Post-transform validation. Must be added to the lint gate pipeline before PR is opened.

### Pitfall 8: Context Migration Re-render Explosion

**What goes wrong:**
Class components using `static contextType` or the legacy `Context.Consumer` pattern are migrated to `useContext`. The migration is syntactically correct, but if the context value is an object or function created inline in the Provider, every re-render of the Provider creates a new reference, causing every `useContext` consumer to re-render even when the logical value has not changed.

This is not a new behavior, but it is invisible until performance profiling. In a large migration, many components may be converted simultaneously, turning a low-frequency problem into a pathological one.

**Prevention:**
- When migrating context consumers, check the Provider call site. If the context value is an inline object or function expression, flag it in the PR description as a performance-review item.
- Do not automatically add `useMemo` around Provider values — this is an architectural decision that may change context semantics.

**Detection (warning signs):**
- Context `value` prop is `{{ ... }}` (inline object) or `() => ...` (inline function)
- Many components consume the same context

**Phase:** PR description generation — surface as an "attention required" note, not a blocker.

### Pitfall 9: PR Noise for Large Codebases — Description Becomes Unreadable

**What goes wrong:**
When migrating 50+ components in a single PR, the agent-generated PR description lists every component, tsc delta, and flag status in a flat format. Reviewers cannot distinguish high-risk components from trivially migrated ones. The PR becomes un-reviewable and is either rubber-stamped or blocked entirely.

**Prevention:**
- Structure the PR description in tiers: "Safe Auto (N components)" as a collapsed summary, "Needs Review (M components)" as an explicit callout list, "Excluded (K components)" with reasons.
- Cap the flat component list at 20 items in any tier; truncate with "and N more" and a link to the full list in a PR comment.
- The PR label `safe-auto-migration` vs `needs-review` should drive the review expectation, not the description length.
- Implement a PR description size budget: if the component count exceeds a threshold (e.g., 30 components), split into multiple PRs by directory or feature area.

**Detection (warning signs):**
- More than 30 components in the migration schedule for a single run
- `assisted` and `manual` components outnumber `safe-auto` components

**Phase:** PR description generation and migration schedule building. The planner (extending `schedule/planner.py`) should enforce a per-PR component count ceiling.

### Pitfall 10: TsMorphProjectEngine Subprocess Diagnostic Scope Mismatch

**What goes wrong:**
`handleInitProject` in `handlers.ts` falls back to hardcoded compiler options (`strict: true`, `ES2022`) when no `tsconfig_path` is provided. This means tsc diagnostics gathered by the migration pipeline may not match what `tsc` reports when run directly in the target project (which has its own `tsconfig.json` with different flags, paths, and includes).

The before/after tsc gate may report zero new errors because the fallback compiler options suppress some checks that the real tsconfig enforces, or vice versa.

**Why it happens:**
The existing `TsMorphProjectEngine.__aenter__` passes `tsconfig_path` only if explicitly provided. There is no auto-discovery of `tsconfig.json` in the workspace root.

**Consequences:**
tsc gate passes in the pipeline but the PR fails CI because the project's real tsc configuration catches errors the pipeline did not.

**Prevention:**
- The migration orchestrator must always discover and pass the workspace `tsconfig.json` to `TsMorphProjectEngine`. Auto-discover: `workspace / "tsconfig.json"`, fallback to `workspace / "tsconfig.base.json"`, hard-fail if neither exists.
- Never use the fallback default compiler options for migration work. The fallback is acceptable for ad-hoc single-file analysis but not for a migration gate.
- Add a startup assertion in the migration runner: verify the engine was initialized with a tsconfig path, not with default options.

**Detection (warning signs):**
- `tsconfig_path` is `None` in `TsMorphProjectEngine` initialization in the migration runner
- Pre-check error count is suspiciously low compared to running `tsc` manually in the workspace

**Phase:** Migration runner initialization. Fix before any tsc gate logic is built.

### Pitfall 11: Bare Exception Handlers in the Existing Engine Swallow Migration Failures

**What goes wrong:**
The codebase already has 20+ bare exception handlers (see `CONCERNS.md`). In the migration path, the most dangerous are in `orchestrator/agent.py` and `a2a/executor.py`. A transform that partially fails (e.g., the subprocess crashes mid-write) may be swallowed, leaving files in a partially migrated state. The migration continues, and the tsc gate sees a corrupted workspace.

**Why it happens:**
Pre-existing technical debt. The bare `except` clauses in the engine subprocess handlers will catch `SubprocessError` and `OSError` but log at the wrong level or not at all.

**Consequences:**
Silent partial migration. tsc gate runs against a partially written workspace. Error count delta is meaningless. Files on the migration branch are in an inconsistent state.

**Prevention:**
- Before building the migration executor, audit and fix bare exception handlers in the subprocess engine path (`engine/subprocess_engine.py`, `agentic/runner.py`).
- The migration executor must implement a transactional model: capture the full set of changed files before applying, and roll back (restore from git) on any unhandled exception. Git branch isolation makes rollback cheap.
- Explicitly handle `SubprocessError` from the engine and surface it as a migration failure, not a silent skip.

**Detection (warning signs):**
- `except Exception:` or bare `except:` in any file in the migration execution path
- `apply_changes()` is called without a surrounding try/except that triggers git rollback

**Phase:** Migration executor design. Address before the executor is built, not after.

## Minor Pitfalls

---

### Pitfall 12: PureComponent / shouldComponentUpdate Naively Replaced with React.memo

**What goes wrong:**
`PureComponent` does shallow comparison on all props and state. `React.memo` does shallow comparison on props only (not state, since function components don't have state in props). If a `PureComponent` was preventing re-renders based on state comparisons, the `React.memo` equivalent will re-render on every state update via hooks.

**Prevention:**
- Document that `PureComponent` → `React.memo` is only correct when the re-render optimization was driven by props, not state.
- Components extending `PureComponent` should be classified as `assisted` to ensure the optimization is validated post-migration.

**Phase:** Classification phase — `PureComponent` extends → `assisted` signal.

---

### Pitfall 13: Legacy String Refs Are Not Detected and Cause Silent Breakage

**What goes wrong:**
Very old React class components may use string refs (`ref="myRef"`, `this.refs.myRef`). These were deprecated in React 16.3 and removed in React 19. ts-morph skeleton inspection may not flag these because string attributes on JSX are syntactically valid. When migrated, the ref silently becomes `undefined`.

**Prevention:**
- Add an explicit check for `this.refs.*` access in the class body and `ref="..."` (string literal) in JSX render output to the pre-migration analysis.
- These components are escalated to `manual` — string refs require a multi-step migration (string ref → callback ref → createRef → useRef) that is too risky to automate.

**Phase:** Classification phase.

---

### Pitfall 14: Module-Level Global State in A2A Executor Creates Race Conditions Under Concurrent Migration Jobs

**What goes wrong:**
`a2a/executor.py` stores session state in module-level dicts (see `CONCERNS.md`). If two migration jobs run concurrently via the A2A endpoint, they share session state, leading to cross-contamination of migration schedules, file lists, and tsc snapshots.

**Prevention:**
- The migration session state (before-snapshot of tsc errors, component schedule, branch name) must be request-scoped, not module-level.
- Use Pydantic model instances passed through the executor call, not global dicts.
- This is a prerequisite: fix before the migration executor is built on top of A2A.

**Phase:** A2A executor refactor — prerequisite before building the migration pipeline on it.

## Phase-Specific Warnings

| Phase Topic | Likely Pitfall | Mitigation |
|-------------|---------------|------------|
| Component classification / triage extension | False-positive `safe-auto` for unmigrateable components | Hard-code blocklist for `getSnapshotBeforeUpdate`, `componentDidCatch`, `forceUpdate` before LLM runs |
| Migration transform (ts-morph) | Hook ordering violations, stale closure in state updates | Post-transform ESLint gate (hooks plugin) before `apply_changes()` |
| tsc before/after gate | Error count gaming, diagnostic scope mismatch | Per-file error-only fingerprints, always pass tsconfig to engine |
| Migration executor | Partial write on subprocess crash | Transactional apply with git rollback on failure |
| PR description generation | Unreadable noise for large migrations | Tiered format, per-PR component count ceiling |
| A2A integration | Concurrent job state contamination | Request-scoped session state, fix before building on A2A |
| getDerivedStateFromProps migration | Timing semantic breakage | Unconditional `assisted` escalation, no automated transform |
| Ref-exposing components | Broken imperative handle contract | Cross-reference `find_references` for `.current.*` call sites |
| React.memo vs PureComponent | State-based optimization lost | `PureComponent` extends → `assisted` signal in classifier |
| tsc strict mode hardening layer (v2) | Enabling all flags at once creates massive error spike | Incremental flag enabling with per-flag before/after gate; one flag per PR |
