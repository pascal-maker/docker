# Architecture Patterns

**Domain:** React class→hooks migration + TypeScript strict mode hardening platform
**Researched:** 2026-04-01
**Overall confidence:** HIGH (based on direct codebase inspection)

---

## Recommended Architecture

The migration capability should be built as a **new `migration/` package** inside
`apps/backend/src/refactor_agent/`, parallel to the existing `agentic/` and `schedule/`
packages. It does not replace any existing pipeline stage — it wraps and extends the
pipeline at well-defined integration points.

### High-Level Flow

```
[MCP tool / CI preset / A2A call]
         │
         ▼
[Router] agentic/router.py — intent: "react_migration" route added
         │
         ▼
[React Classifier]  migration/classifier.py          ← NEW
   • Calls TsMorphProjectEngine.get_skeleton() per .tsx file
   • Calls get_diagnostics() to establish baseline tsc error count
   • Classifies each class component: simple / medium / complex / skip
   • Returns MigrationScope (replaces ScopeSpec for migration path)
         │
         ▼
[Migration Planner]  migration/planner.py            ← NEW
   • Builds MigrationSchedule: ordered list of MigrateComponentOp
   • Dependency ordering: if ComponentA renders ComponentB, B migrates first
   • Three tiers: safe_auto | assisted | manual (excluded)
         │
         ▼
[tsc Before-Gate]   migration/tsc_gate.py            ← NEW
   • Calls TsMorphProjectEngine.get_diagnostics(file_path=None) → baseline count
   • Stores TscGateResult(before_errors: list[DiagnosticInfo], count: int)
   • Hard-blocks execution if workspace is already broken past threshold
         │
         ▼
[Migration Executor]  migration/executor.py          ← NEW
   • Calls agentic/git.py::ensure_refactor_branch() — reuses existing
   • Per component: calls ts-morph-bridge via new handlers (convert_class_component)
   • On each component success: calls agentic/git.py::commit_checkpoint()
   • On failure: calls agentic/git.py::reset_to_last_commit() + marks component manual
         │
         ▼
[tsc After-Gate]    migration/tsc_gate.py            ← NEW
   • Calls get_diagnostics(file_path=None) again → after_errors count
   • Computes delta = after_errors - before_errors
   • If delta > 0 and no override: sets MigrationResult.tsc_gate_failed = True
   • PR is NOT opened; error surfaced to caller
         │
         ▼
[PR Creator]        migration/pr.py                  ← NEW
   • Shells out to `gh pr create` via subprocess (same pattern as agentic/git.py)
   • PR body generated by PRDescriptionAgent (PydanticAI agent, low-token task)
   • Applies labels: safe-auto-migration | needs-review
         │
         ▼
[MigrationResult]   migration/models.py              ← NEW
   Returned to caller (MCP tool / CI report / A2A response)
```

## Component Boundaries

| Component | File | Responsibility | Communicates With |
|-----------|------|---------------|-------------------|
| `ReactClassifier` | `migration/classifier.py` | Scan workspace, classify components by migration tier | `TsMorphProjectEngine`, `MigrationScope` model |
| `MigrationScope` | `migration/models.py` | Replaces `ScopeSpec` for migration; holds component list with tiers | All migration stages |
| `MigrationSchedule` | `migration/models.py` | Ordered `MigrateComponentOp` list with dependency graph | `MigrationExecutor` |
| `TscGate` | `migration/tsc_gate.py` | Before/after error snapshot; delta computation | `TsMorphProjectEngine.get_diagnostics()` |
| `MigrationExecutor` | `migration/executor.py` | Branch, per-component transform, commit/reset | `agentic/git.py`, `TsMorphProjectEngine`, ts-morph-bridge |
| `PRCreator` | `migration/pr.py` | `gh pr create` subprocess + PR description agent | `agentic/git.py`, `PRDescriptionAgent` |
| `MigrationRunner` | `migration/runner.py` | Orchestrates classifier → planner → tsc-gate → executor → pr | All migration components |
| New ts-morph handlers | `packages/ts-morph-bridge/src/handlers.ts` | `convert_class_component` JSON-RPC method | Called from `MigrationExecutor` via `TsMorphProjectEngine` |

## Integration Points with Existing Architecture

### 1. Router (`agentic/router.py`)

The existing router returns a `RouteResult` with intent `refactor | general_chat`. A
new intent value `react_migration` should be added to the `RouteResult` discriminator.
When this intent is detected, `agentic/runner.py` dispatches to `MigrationRunner` instead
of the standard triage → planner → executor path.

**Exact file:** `apps/backend/src/refactor_agent/agentic/router.py`
The `RouteResult` model adds `"react_migration"` as a valid intent literal.

### 2. Triage Classification (`agentic/triage.py`)

The migration classifier does NOT extend `run_triage()` directly. The existing triage
categories (`trivial | structural | paradigm_shift | ambiguous`) do not map cleanly to
React migration tiers (`simple | medium | complex`). Instead:

- The `ReactClassifier` is a separate PydanticAI agent with its own output type
  `ComponentClassification` (new Pydantic model).
- It reuses the same triage **tools** (`list_workspace_files`, `show_file_skeleton`,
  `find_references`) by calling `TsMorphProjectEngine` directly — the same pattern
  already used in `agentic/triage.py` lines 79–157.
- `OrchestratorDeps` is passed in unchanged — no new deps container needed.

**Exact file:** `apps/backend/src/refactor_agent/agentic/triage.py` (read-only reference)

### 3. ScopeSpec → MigrationScope

`ScopeSpec` (`schedule/models.py` lines 82–87) constrains file paths and op types for
the existing refactor pipeline. The migration path needs richer metadata per component
(lifecycle methods present, state shape, refs, tier). Rather than cramming this into
`ScopeSpec`:

- Define `MigrationScope(BaseModel)` in `migration/models.py` that carries
  `components: list[ComponentClassification]` and `baseline_tsc_errors: int`.
- `MigrationScope` is the "scope envelope" for the migration pipeline, analogous to
  `ScopeSpec` in the refactor pipeline.

**Exact file:** `apps/backend/src/refactor_agent/schedule/models.py` (read-only; ScopeSpec
is not modified — migration has its own scope model)

### 4. TscGate plugs into `execute_schedule()` lifecycle

`execute_schedule()` in `schedule/executor.py` has no before/after hooks. The migration
path does NOT modify `execute_schedule()`. Instead:

- `MigrationExecutor.run()` calls `TscGate.snapshot_before()` → runs per-component
  transforms via `MigrationExecutor._apply_one_component()` → calls
  `TscGate.snapshot_after()` → computes delta.
- `TscGate` calls `TsMorphProjectEngine.get_diagnostics(file_path=None)` which already
  returns the full project diagnostic list (`engine/base.py` line 168;
  `engine/typescript/ts_morph_engine.py` lines 261–285).
- This is already wired through the JSON-RPC bridge: `handlers.ts` exports
  `handleGetDiagnostics` and `index.ts` maps `"get_diagnostics"` to it.

**Confidence:** HIGH — `get_diagnostics` is fully implemented end-to-end.

### 5. Git Branch and Commit (`agentic/git.py`)

`agentic/git.py` already provides:
- `ensure_refactor_branch(workspace, goal)` → creates `refactor/<slug>-<ts>` branch
- `commit_checkpoint(workspace, message)` → `git add -A && git commit`
- `reset_to_last_commit(workspace)` → `git reset --hard HEAD`

`MigrationExecutor` calls these directly — no new git primitives needed for v1.
PR branch naming should use `migration/react-hooks-<ts>` rather than `refactor/` prefix,
which means `ensure_refactor_branch()` should either accept a prefix parameter or
`migration/executor.py` calls `git checkout -b` directly with the migration prefix.

**Exact file:** `apps/backend/src/refactor_agent/agentic/git.py`
**Gap:** `ensure_refactor_branch()` hard-codes `refactor/` prefix (line 46). Either add
a `branch_prefix: str = "refactor"` parameter or the migration module calls git directly.

### 6. PR Creation (`migration/pr.py`)

The existing codebase has no PR creation. New module follows the same subprocess pattern
as `agentic/git.py`:

```python
subprocess.run(
    ["gh", "pr", "create", "--title", title, "--body", body,
     "--label", label, "--head", branch],
    cwd=workspace,
    check=True,
    capture_output=True,
    timeout=30,
)
```

`gh` CLI must be present and authenticated in the execution environment.
The PR description is generated by `PRDescriptionAgent` — a minimal PydanticAI agent
that takes `MigrationResult` as input and returns a formatted markdown string. This
keeps LLM logic in agents, not in infrastructure modules.

### 7. MCP Server (`mcp/server.py`)

Currently only exposes `rename_symbol`. New migration tools to add:
- `run_react_migration(workspace_path, tier_filter)` — triggers `MigrationRunner`
- `get_migration_plan(workspace_path)` — dry-run: returns plan without executing

**Exact file:** `apps/backend/src/refactor_agent/mcp/server.py`

### 8. ts-morph-bridge New Handler (`packages/ts-morph-bridge/src/handlers.ts`)

The existing bridge has no React class→hooks transformation. A new handler is needed:

```typescript
export function handleConvertClassComponent(
  params: Record<string, unknown>
): { summary: string; changed: boolean }
```

This handler receives `file_path` and `component_name`. It uses ts-morph AST APIs to:
1. Detect `extends React.Component` / `extends Component` via `sf.getClasses()`
2. Extract `render()` method body → function component body
3. Map lifecycle methods: `componentDidMount` → `useEffect(fn, [])`,
   `componentDidUpdate(prevProps, prevState)` → `useEffect` with dependency array,
   `componentWillUnmount` → `useEffect` cleanup return
4. Map `this.state` → `useState` hooks, `this.setState` → setter calls
5. Map `this.props` → destructured props parameter
6. Write the new function component back via `sf.replaceText()`

**Exact file:** `packages/ts-morph-bridge/src/handlers.ts`
**Also register in:** `packages/ts-morph-bridge/src/index.ts` handlers map (line 31–52)

## Data Flow: Full Migration Run

```
MigrationRunner.run(goal, deps)
  │
  ├─ ReactClassifier.classify(deps)
  │    └─ TsMorphProjectEngine.get_skeleton() per .tsx file
  │    └─ returns MigrationScope(components=[...], baseline_tsc_errors=N)
  │
  ├─ MigrationPlanner.build_schedule(scope)
  │    └─ returns MigrationSchedule (toposorted MigrateComponentOp list)
  │
  ├─ TscGate.snapshot_before(deps)
  │    └─ TsMorphProjectEngine.get_diagnostics() → TscSnapshot(errors=N)
  │
  ├─ agentic/git.py::ensure_refactor_branch(workspace, goal)
  │
  ├─ MigrationExecutor.execute(schedule, deps)
  │    for each component in topological order:
  │      ├─ TsMorphProjectEngine._call("convert_class_component", {file, component})
  │      ├─ TsMorphProjectEngine.apply_changes()
  │      └─ agentic/git.py::commit_checkpoint(workspace, f"migrate: {component}")
  │
  ├─ TscGate.snapshot_after(deps)
  │    └─ TsMorphProjectEngine.get_diagnostics() → TscSnapshot(errors=M)
  │    └─ delta = M - N; gate fails if delta > 0
  │
  ├─ [if gate passes] PRCreator.create_pr(branch, result)
  │    └─ PRDescriptionAgent.generate(result) → markdown body
  │    └─ subprocess: gh pr create ...
  │
  └─ returns MigrationResult
```

## Models to Create (`migration/models.py`)

All Pydantic `BaseModel`, no `dict` in signatures, strict typing per CLAUDE.md.

```python
class MigrationTier(StrEnum):
    SAFE_AUTO = "safe_auto"
    ASSISTED = "assisted"
    MANUAL = "manual"

class ComponentClassification(BaseModel):
    file_path: str
    component_name: str
    tier: MigrationTier
    lifecycle_methods: list[str]
    has_state: bool
    has_refs: bool
    has_error_boundary: bool
    rationale: str | None = None

class MigrationScope(BaseModel):
    components: list[ComponentClassification]
    baseline_tsc_errors: int
    workspace: str  # str not Path — Pydantic serialization

class MigrateComponentOp(BaseModel):
    op: Literal["migrate_component"] = "migrate_component"
    id: str | None = None
    depends_on: list[str] = Field(default_factory=list)
    file_path: str
    component_name: str
    tier: MigrationTier

class MigrationSchedule(BaseModel):
    goal: str
    operations: list[MigrateComponentOp]
    scope: MigrationScope

class TscSnapshot(BaseModel):
    error_count: int
    errors: list[DiagnosticInfo]  # from engine/base.py

class TscGateResult(BaseModel):
    before: TscSnapshot
    after: TscSnapshot
    delta: int
    passed: bool

class MigrationResult(BaseModel):
    success: bool
    branch: str
    pr_url: str | None = None
    components_migrated: int
    components_skipped: int
    tsc_gate: TscGateResult | None = None
    error: str | None = None
```

## Build Order / Dependency Graph

Dependencies run top-to-bottom. Each item must be buildable before the item that
depends on it.

```
1. migration/models.py
   — no upstream deps beyond engine/base.py (already exists)

2. packages/ts-morph-bridge/src/handlers.ts  (handleConvertClassComponent)
   — needed before migration/executor.py can call it
   — register in index.ts handlers map simultaneously

3. engine/typescript/ts_morph_engine.py  (add convert_class_component method)
   — thin Python wrapper over new bridge handler
   — follows exact pattern of existing rename_symbol / remove_node methods

4. migration/tsc_gate.py
   — uses TsMorphProjectEngine.get_diagnostics() (already exists)
   — uses TscSnapshot, TscGateResult models from step 1

5. agentic/git.py  (add branch_prefix parameter to ensure_refactor_branch)
   — small extension to existing function, no new module

6. migration/classifier.py
   — uses TsMorphProjectEngine.get_skeleton(), get_diagnostics() (already exist)
   — uses ComponentClassification, MigrationScope from step 1

7. migration/planner.py
   — uses ComponentClassification, MigrationSchedule from step 1
   — pure logic: topological sort on component render-graph
   — no LLM needed for v1 (heuristic ordering); LLM optional for complex cases

8. migration/executor.py
   — uses convert_class_component from step 3
   — uses agentic/git.py from step 5
   — uses TscGate from step 4

9. migration/pr.py
   — uses agentic/git.py (already exists)
   — requires gh CLI available in environment

10. migration/runner.py
    — orchestrates steps 6 → 7 → 4 → 8 → 9
    — returns MigrationResult

11. agentic/router.py  (add react_migration intent)
    — wires migration intent to MigrationRunner

12. mcp/server.py  (add run_react_migration, get_migration_plan tools)
    — surface migration to Claude Code / Cursor

13. ci/runner.py  (add migration preset type)
    — optional; allows CI-triggered migrations
```

## Anti-Patterns to Avoid

### Anti-Pattern 1: Modifying execute_schedule() to add tsc gate

**What it looks like:** Adding `before_tsc_check` / `after_tsc_check` parameters to
`schedule/executor.py::execute_schedule()`.

**Why bad:** `execute_schedule()` is the shared execution path for all refactor types.
Adding migration-specific concerns pollutes it, breaks the single-responsibility boundary,
and risks introducing regressions in non-migration flows.

**Instead:** `MigrationExecutor` is a separate executor that calls `execute_schedule()`
only if the migration shares op types, or manages its own execution loop for
`MigrateComponentOp` which is not a `RefactorOperation` subtype.

### Anti-Pattern 2: Using jscodeshift alongside ts-morph

**What it looks like:** Adding a jscodeshift pipeline for the React transforms because
jscodeshift has pre-built codemods.

**Why bad:** The project constraint (PROJECT.md line 62) explicitly states "do not
introduce a second TS engine." `TsMorphProjectEngine` is already the execution backbone.
jscodeshift codemods are also typically untyped and produce no guarantees compatible with
tsc-gate verification.

**Instead:** Implement `handleConvertClassComponent` in the existing ts-morph-bridge.
ts-morph has full access to the TypeScript compiler API and can detect class components,
lifecycle methods, and state shape with full type information.

### Anti-Pattern 3: Extending ScopeSpec for migration metadata

**What it looks like:** Adding `migration_tier`, `lifecycle_methods`, `component_name`
fields to the existing `ScopeSpec` in `schedule/models.py`.

**Why bad:** `ScopeSpec` is consumed by `agentic/validator.py` and all existing triage
paths. Adding migration-specific fields breaks the refactor-pipeline contract and creates
a model that tries to serve two different workflows.

**Instead:** `MigrationScope` is a separate model (see models section above) that
provides the scope envelope specifically for migration orchestration.

### Anti-Pattern 4: Generating the PR description from `migration/pr.py` directly

**What it looks like:** String-formatting the PR description in Python infrastructure code
without an agent.

**Why bad:** The PR description is described in PROJECT.md as "the demo moment" — it
needs to be high quality and readable. Hardcoded templates degrade over time.

**Instead:** `PRDescriptionAgent` is a minimal PydanticAI agent whose single job is
formatting a `MigrationResult` into markdown. Cheap (low token count), testable with
`TestModel`, and consistent with the project's pattern of keeping all LLM logic inside
agents.

## Scalability Considerations

| Concern | 10 components | 100 components | 1000+ components |
|---------|--------------|----------------|-----------------|
| Classification time | Single TsMorphProjectEngine session, fast | Still one session, O(n) skeleton calls | Batch by directory; stream results |
| tsc before/after | Single project diagnostic call (already O(project)) | Same | Same — ts-morph loads whole project once |
| Git branch | One branch per run | One branch per run | Consider per-batch branches |
| PR description | Single PRDescriptionAgent call | Same | May need pagination of component list |
| Bridge subprocess | One process per TsMorphProjectEngine context manager | Same pattern | Long-running process pool if needed (future) |

## Where Strict Mode Hardening (Layer 2) Fits

Layer 2 (TypeScript strict mode hardening) runs after the hooks migration. It does not
require a new pipeline — it reuses the existing `RefactorSchedule` + `execute_schedule()`
path with a new CI preset type (`strict_mode_hardening`). The tsc gate is the same
`TscGate` component. The key difference is that strict mode hardening enables compiler
flags incrementally (one flag at a time) and measures the tsc error delta per flag,
rather than per component.

Layer 2 is a Phase 2+ concern and does not need to be built before the React migration
path is functional.
